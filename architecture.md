# معماری

## مقدمه

این معماری یک ساختار ماژولار و چندسکویی برای توسعه محصولات نرم‌افزاری مبتنی بر JavaScript، HTML و CSS است که بدون وابستگی به چارچوب‌های سنگین طراحی شده است.

هدف اصلی این معماری کاهش پیچیدگی، افزایش قابلیت نگهداری، بهبود استفاده مجدد از کد، و ساده‌سازی توسعه توسط انسان و مدل‌های هوش مصنوعی است.

این معماری تلاش می‌کند وابستگی به چارچوب‌ها، رفتارهای پنهان، و انتزاع‌های پیچیده را کاهش دهد و در عوض بر قراردادهای شفاف، مرزبندی مشخص، و ساختارهای قابل‌پیش‌بینی تکیه کند.

## اهداف معماری

معماری باید:

* توسعه محصول را سریع‌تر کند
* استفاده مجدد از قابلیت‌ها را افزایش دهد
* وابستگی میان بخش‌های محصول را کاهش دهد
* توسعه توسط انسان و مدل‌های هوش مصنوعی را ساده‌تر و قابل‌اعتمادتر کند
* در بلندمدت بدون بازنویسی کامل تکامل پیدا کند

## ساختار کلی

سامانه از پنج بخش اصلی تشکیل شده است که هر کدام مسئولیت مشخصی دارند:

```text
        ماژول‌ها
       ↙         ↘
رابط کاربری    افزونه‌ها
       ↘         ↙
          هسته
            ↓
          پلتفرم
```

* **ماژول‌ها** منطق و قابلیت‌های اصلی محصول را پیاده‌سازی می‌کنند
* **رابط کاربری** داده را نمایش می‌دهد و تعاملات کاربر را دریافت می‌کند
* **افزونه‌ها** زیرساخت‌های قابل استفاده مشترک را فراهم می‌کنند
* **هسته** هماهنگی میان بخش‌ها را از طریق گذرگاه رویداد مدیریت می‌کند
* **پلتفرم** اتصال به محیط اجرا را فراهم می‌کند

## لایه‌های معماری

### هسته

هسته بخش مرکزی معماری است که مسئول هماهنگی و مدیریت اجزای معماری است.

هسته منطق پروژه را نمی‌شناسد، وابسته به رابط کاربری نیست، و از جزئیات محیط اجرا آگاه نیست. هسته در انتقال داده میان بخش‌ها دخالتی ندارد.

وظایف هسته:

* مدیریت چرخه حیات سیستم
* بارگذاری و مدیریت چرخه حیات بخش‌ها
* اعتبارسنجی مانیفست و قرارداد
* رجیستری سرویس‌ها
* مدیریت گذرگاه‌های رویداد
* مدیریت خطا
* نگهداری حالت مشترک
* مدیریت پیکربندی
* لاگ داخلی

### گذرگاه رویداد

گذرگاه رویداد تنها کانال مجاز برای اطلاع‌رسانی میان بخش‌هاست و توسط هسته مدیریت می‌شود.

```text
گذرگاه رویداد  →  اطلاع‌رسانی + داده حداقلی   "چه اتفاقی افتاد و روی چه چیزی"
واسط ماژول     →  انتقال داده کامل       "داده کامل را بده"
```

رویداد می‌تواند یک فیلد `data` اختیاری همراه داشته باشد که فقط شامل اطلاعات حداقلی و ضروری است. داده کامل هرگز در آن قرار نمی‌گیرد. داده‌های حساس مثل توکن یا رمز عبور نباید در این فیلد قرار بگیرند.

```text
task:created  →  { data: { id: "123" } }
file:uploaded →  { data: { id: "abc123", size: 10485760, mimeType: "image/png" } }
core:ready    →  بدون data
```

### افزونه‌ها

افزونه‌ها زیرساخت‌ها و قابلیت‌های مشترک میان پروژه‌ها را فراهم می‌کنند. افزونه‌ها از روز اول بخشی از سیستم هستند و پایه‌ای برای کار ماژول‌ها محسوب می‌شوند.

افزونه‌ها مسئول فراهم‌کردن زیرساخت هستند، نه پیاده‌سازی منطق محصول.

نمونه‌ها:

* ذخیره‌سازی
* احراز هویت
* مسیریابی
* ارتباط با هوش مصنوعی
* تحلیل داده

### ماژول‌ها

ماژول‌ها ویژگی‌ها و منطق اصلی محصول را پیاده‌سازی می‌کنند. هر ماژول نماینده یک قابلیت کسب‌وکار یا ویژگی مستقل است.

نمونه‌ها:

* مدیریت وظایف
* گفت‌وگو
* پروژه‌ها
* اعلان‌ها

ماژول‌ها می‌توانند از افزونه‌ها استفاده کنند، اما نباید مستقیماً به یکدیگر وابسته باشند. هر ماژول یک واسط عمومی دارد که داده از طریق آن در دسترس قرار می‌گیرد.

### رابط کاربری

لایه رابط کاربری مسئول نمایش داده و دریافت تعاملات کاربر است.

رابط کاربری نباید شامل منطق اصلی باشد. منطق کسب‌وکار باید خارج از رابط کاربری قرار گیرد تا قابلیت آزمایش، استفاده مجدد، و اجرا در محیط‌های مختلف داشته باشد.

رابط کاربری از طریق گذرگاه رویداد از تغییرات مطلع می‌شود و داده را مستقیماً از واسط ماژول‌ها دریافت می‌کند:

```text
ماژول پیام منتشر می‌کند  →  رابط کاربری مطلع می‌شود  →  داده را از واسط ماژول می‌خواهد  →  نمایش به‌روز می‌شود
```

### پلتفرم

لایه پلتفرم مسئول اتصال محصول به محیط اجرا است و امکان اجرای محصول در محیط‌های مختلف را فراهم می‌کند.

نمونه‌ها:

* مرورگر
* دسکتاپ
* موبایل

تمام دسترسی‌ها به قابلیت‌های محیط اجرا باید از طریق واسط‌ها و قراردادهای مشخص انجام شود.

## اصول بنیادین

### ۱. سادگی بر پیچیدگی مقدم است

سادگی یک انتخاب تزئینی نیست، بلکه یک اصل معماری است. هر بخش از محصول باید قابل‌درک، رفتار قابل‌پیش‌بینی، و بدون نیاز به دانش پنهان قابل توسعه باشد.

پیچیدگی تنها زمانی پذیرفته می‌شود که مزیت واقعی و بلندمدت ایجاد کند.

### ۲. منطق کسب‌وکار باید مستقل باشد

منطق محصول نباید به رابط کاربری، نوع پایگاه داده، محیط اجرا یا سرویس‌های خارجی وابسته باشد و باید بتواند بدون تغییر در محیط‌های مختلف اجرا شود.

### ۳. همه‌چیز باید از طریق قراردادها ارتباط برقرار کند

اجزای محصول نباید مستقیماً به پیاده‌سازی یکدیگر وابسته باشند. ارتباط میان بخش‌ها باید تنها از طریق قراردادهای مشخص و پایدار انجام شود.

قراردادها پایه اصلی توسعه‌پذیری، قابلیت تعویض، آزمون‌پذیری، و توسعه توسط هوش مصنوعی هستند.

### ۴. هسته باید کوچک و ناآگاه باقی بماند

هسته نباید منطق پروژه، رابط کاربری، یا جزئیات محیط اجرا را بشناسد. هرچه هسته کوچک‌تر و عمومی‌تر باشد، محصول پایدارتر و قابل‌استفاده‌تر خواهد بود.

### ۵. همه قابلیت‌ها باید قابل‌تعویض باشند

هیچ بخش از محصول نباید به یک پیاده‌سازی خاص وابسته شود. سیستم ذخیره‌سازی، احراز هویت، موتور رابط کاربری، یا روش ارتباط با API باید بدون تغییر در منطق اصلی قابل جایگزینی باشند.

### ۶. ماژول‌ها باید مستقل و ایزوله باشند

هر ماژول باید مسئولیت مشخصی داشته باشد، مرز واضحی تعریف کند، و بدون وابستگی مستقیم به سایر ماژول‌ها توسعه یابد.

### ۷. رابط کاربری فقط لایه نمایش است

وظیفه رابط کاربری نمایش داده، دریافت تعامل کاربر، و ارسال پیام‌هاست. تمام تصمیم‌ها و قوانین اصلی باید خارج از رابط کاربری قرار بگیرند.

### ۸. معماری باید برای انسان و هوش مصنوعی قابل‌فهم باشد

این معماری برای مدل‌های هوش مصنوعی نیز باید قابل‌تحلیل و قابل‌توسعه باشد. به همین دلیل محصول باید ساختارهای واضح، فایل‌های کوچک، وابستگی محدود، رفتارهای شفاف، و قراردادهای مشخص داشته باشد.

### ۹. ایزوله‌سازی خطا

خرابی یا نقص در یک ماژول یا افزونه نباید باعث از کار افتادن کل سیستم شود. خطاها باید در همان مرزی که رخ می‌دهند مهار شوند و سیستم باید بتواند به کار خود، حتی با کاهش قابلیت، ادامه دهد.

### ۱۰. مالکیت صریح داده‌ها

هر قطعه از داده در سیستم باید یک مالک مشخص و واحد داشته باشد. هیچ بخشی نباید داده بخش دیگر را به صورت مستقیم تغییر دهد. درخواست برای تغییر داده باید از طریق قراردادها و پیام‌ها انجام شود.

### ۱۱. انتقال داده باید صریح و مستقیم باشد

داده کامل فقط از طریق واسط عمومی ماژول‌ها منتقل می‌شود. ماژول‌ها داده یکدیگر را مستقیم نمی‌خوانند. رابط کاربری داده را از ماژول‌ها می‌خواهد، نه برعکس. گذرگاه رویداد فقط برای اطلاع‌رسانی است و می‌تواند داده حداقلی همراه داشته باشد.

## قوانین ارتباط میان بخش‌ها

جهت وابستگی‌ها در محصول باید مشخص و کنترل‌شده باشد:

```text
ماژول‌ها ← افزونه‌ها ← هسته ← پلتفرم
```

* هسته نباید به افزونه‌ها، ماژول‌ها، یا پروژه خاص وابسته باشد
* ماژول‌ها می‌توانند از افزونه‌ها استفاده کنند، اما نباید مستقیماً به یکدیگر وابسته باشند
* ماژول‌ها فقط از طریق گذرگاه رویداد با یکدیگر اطلاع‌رسانی می‌کنند
* تمام ارتباطات باید از طریق قراردادهای مشخص انجام شود
* هیچ بخشی نباید مستقیماً به پیاده‌سازی داخلی بخش دیگر وابسته باشد

## هدف نهایی

هدف این معماری ساخت محصولی است که توسعه آن سریع، نگهداری آن ساده، وابستگی کمی داشته باشد، در محیط‌های مختلف اجرا شود، و بتواند در بلندمدت بدون بازنویسی کامل تکامل پیدا کند.

این معماری تلاش می‌کند تعادلی میان سادگی، انعطاف‌پذیری، توسعه‌پذیری و پایداری بلندمدت ایجاد کند.

## اصول تست‌پذیری

تست‌پذیری در این معماری یک اصل است، نه یک انتخاب. معماری باید به گونه‌ای طراحی شود که هر بخش بتواند مستقل از بقیه سیستم تست شود.

### اصول پایه

* هر بخش باید بدون نیاز به هسته قابل تست باشد
* هر بخش باید بتوان وابستگی‌هایش را با نمونه‌های جعلی جایگزین کرد
* قراردادها پایه تست‌پذیری هستند؛ اگر قرارداد مشخص باشد، تست نوشتن ساده است
* منطق داخلی هر بخش باید مستقل از محیط اجرا قابل تست باشد

### تست قرارداد

هر پیاده‌سازی باید در برابر قرارداد خود تست شود. این یعنی:

* همه متدهای اعلام‌شده در قرارداد باید تست داشته باشند
* تست‌ها باید رفتار اعلام‌شده را، نه جزئیات پیاده‌سازی را، بررسی کنند
* اگر قرارداد تغییر کند، تست‌ها باید به‌روز شوند

## وابستگی به کتابخانه‌های خارجی

وابستگی به کتابخانه‌های خارجی باید کنترل‌شده و محدود باشد. هر کتابخانه خارجی یک وابستگی است که نگهداری، به‌روزرسانی، و جایگزینی آن را پیچیده‌تر می‌کند.

قوانین وابستگی برای هر بخش:

**هسته** — هیچ کتابخانه خارجی نباید داشته باشد. هسته باید کاملاً مستقل باشد تا در هر محیطی بدون وابستگی اضافه اجرا شود.

**افزونه‌ها** — می‌توانند کتابخانه خارجی داشته باشند چون زیرساخت هستند و مسئول پیاده‌سازی قابلیت‌های تخصصی‌اند. وابستگی‌های خارجی افزونه باید در مانیفست اعلام شوند.

**ماژول‌ها** — نباید مستقیماً به کتابخانه خارجی وابسته باشند. اگر ماژولی به قابلیتی از یک کتابخانه خارجی نیاز دارد، باید آن قابلیت از طریق یک افزونه فراهم شود.

**پلتفرم** — می‌تواند کتابخانه خارجی داشته باشد چون به قابلیت‌های محیط اجرا وابسته است.

**رابط کاربری** — می‌تواند کتابخانه‌های نمایشی داشته باشد، اما نباید از کتابخانه‌هایی استفاده کند که منطق کسب‌وکار دارند.

این قوانین از سه اصل معماری پیروی می‌کنند: استقلال بخش‌ها، قابلیت تعویض، و کوچک نگه داشتن هسته.
