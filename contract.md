# قرارداد

## مقدمه

قرارداد یک توافق رسمی است که مشخص می‌کند یک بخش از سیستم چه متدهایی ارائه می‌دهد، چه پیام‌هایی منتشر می‌کند، و از کدام مجرا استفاده می‌کند. قرارداد زبان مشترک میان هسته، افزونه‌ها، ماژول‌ها، پلتفرم، و رابط کاربری است.

قرارداد با مانیفست فرق دارد. مانیفست مشخص می‌کند یک بخش چه کسی است و به چه چیزی نیاز دارد. قرارداد مشخص می‌کند یک بخش چه چیزی ارائه می‌دهد و چگونه ارتباط برقرار می‌کند.

بدون قرارداد، هر بخش باید از جزئیات داخلی بخش دیگری آگاه باشد. با قرارداد، هر بخش فقط باید بداند طرف مقابل چه وعده‌ای داده است.

## فرمت قرارداد

قرارداد یک فایل JSON است با نام `contract.json`. هسته این فایل را همزمان با مانیفست می‌خواند و اعتبارسنجی می‌کند.

## ساختار یک قرارداد

هر قرارداد از پنج بخش تشکیل می‌شود:

**شناسه (identity)** — شامل نام، توضیحات، و نسخه قرارداد. نام شناسه یکتای این قرارداد در سیستم است. هیچ دو قراردادی نمی‌توانند نام یکسان داشته باشند.

**متدها (methods)** — فهرست متدهای عمومی که این بخش ارائه می‌دهد. هر متد باید نام، توضیح، پارامترهای ورودی، نوع خروجی، و خطاهای احتمالی مشخصی داشته باشد.

**پیام‌ها (messages)** — فهرست پیام‌هایی که این بخش منتشر می‌کند. هر پیام باید نوع، توضیح، ساختار داده، و زمان ارسال مشخصی داشته باشد.

**مجرای پیشنهادی (suggestedChannel)** — شناسه مجرایی که پیشنهاد می‌شود برای پیام‌های این بخش استفاده شود.

## اعتبارسنجی قرارداد

هسته اعتبارسنجی قرارداد را در دو مرحله انجام می‌دهد:

**مرحله اول — اعتبارسنجی ساختار** (قبل از بارگذاری بخش):
```text
آیا فیلدهای شناسه، متدها، و پیام‌ها وجود دارند؟
آیا نام قرارداد یکتاست؟
آیا نسخه فرمت درستی دارد؟ (MAJOR.MINOR)
آیا هر متد نام، پارامترها، و نوع خروجی دارد؟
آیا هر پیام نوع، ساختار داده، و زمان ارسال دارد؟
آیا مجرای پیشنهادی مشخص شده است؟
```

**مرحله دوم — اعتبارسنجی پیاده‌سازی** (بعد از بارگذاری بخش):
```text
آیا همه متدهای اعلام‌شده پیاده‌سازی شده‌اند؟
آیا امضای هر متد با قرارداد مطابقت دارد؟
```

اگر اعتبارسنجی ساختار شکست بخورد، بخش بارگذاری نمی‌شود. اگر اعتبارسنجی پیاده‌سازی شکست بخورد، بخش از رجیستری حذف می‌شود و هسته خطای واضحی اعلام می‌کند.

## نمونه قرارداد افزونه

```json
{
  "identity": {
    "name": "my-company.contract.storage",
    "description": "قرارداد ذخیره‌سازی عمومی",
    "version": "1.0"
  },
  "methods": [
    {
      "name": "get",
      "description": "دریافت یک مقدار ذخیره شده",
      "params": [
        { "name": "key", "type": "string" }
      ],
      "returns": "any",
      "errors": ["STORAGE_NOT_FOUND", "STORAGE_FAILED"]
    },
    {
      "name": "set",
      "description": "ذخیره یک مقدار",
      "params": [
        { "name": "key", "type": "string" },
        { "name": "value", "type": "any" }
      ],
      "returns": "boolean",
      "errors": ["STORAGE_FAILED"]
    },
    {
      "name": "delete",
      "description": "حذف یک مقدار ذخیره شده",
      "params": [
        { "name": "key", "type": "string" }
      ],
      "returns": "boolean",
      "errors": ["STORAGE_FAILED"]
    }
  ],
  "messages": [
    {
      "type": "storage:ready",
      "description": "افزونه آماده استفاده است",
      "data": {},
      "when": "پس از راه‌اندازی موفق"
    },
    {
      "type": "storage:synced",
      "description": "داده‌ها همگام‌سازی شدند",
      "data": { "keys": "string[]" },
      "when": "پس از هر عملیات نوشتن"
    },
    {
      "type": "storage:failed",
      "description": "عملیات ذخیره‌سازی شکست خورد",
      "data": { "code": "string" },
      "when": "هنگام بروز خطا"
    }
  ],
  "suggestedChannel": "system.bus"
}
```

## قوانین قرارداد

### تعریف قرارداد

* هر ماژول، افزونه، پلتفرم، و رابط کاربری باید یک فایل `contract.json` داشته باشد
* قرارداد باید پیش از پیاده‌سازی نوشته شود
* قرارداد متعلق به بخشی است که آن را تعریف کرده، نه بخشی که از آن استفاده می‌کند
* هسته همیشه قرارداد و مانیفست را با هم می‌خواند؛ نبود هر کدام خطاست

### پیام‌ها و داده

* پیام‌ها می‌توانند یک فیلد `data` همراه داشته باشند
* داده‌های کوچک می‌توانند مستقیماً در پیام منتقل شوند
* داده‌های حجیم باید در یک افزونه ذخیره‌سازی موقت نگه داشته شوند و فقط شناسه آن در پیام منتقل شود
* داده‌های حساس مثل توکن یا رمز عبور نباید در پیام قرار بگیرند
* مجرای پیشنهادی فقط یک پیشنهاد است و توسعه‌دهنده می‌تواند در پیکربندی پروژه آن را تغییر دهد

### تغییر قرارداد

* تغییری که متد یا پیام جدید اضافه کند بدون تغییر موارد موجود، نسخه فرعی را افزایش می‌دهد
* تغییری که امضای متد موجود را بشکند یا ساختار داده پیام را تغییر دهد، نسخه اصلی را افزایش می‌دهد
* هیچ بخشی نمی‌تواند قرارداد بخش دیگری را تغییر دهد

## آنچه قرارداد نیست

* قرارداد شامل جزئیات پیاده‌سازی نمی‌شود
* قرارداد توضیح نمی‌دهد چطور کاری انجام می‌شود، فقط چه کاری انجام می‌شود
* قرارداد شامل وابستگی‌ها یا پیکربندی نمی‌شود؛ این‌ها در مانیفست تعریف می‌شوند
