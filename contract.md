# قرارداد

## مقدمه

قرارداد یک توافق رسمی است که مشخص می‌کند یک بخش از سیستم چه متدهایی ارائه می‌دهد. قرارداد زبان مشترک میان هسته، افزونه‌ها، ماژول‌ها، پلتفرم، و رابط کاربری است.

قرارداد با مانیفست فرق دارد. مانیفست مشخص می‌کند یک بخش چه کسی است و به چه چیزی نیاز دارد. قرارداد مشخص می‌کند یک بخش چه چیزی ارائه می‌دهد.

بدون قرارداد، هر بخش باید از جزئیات داخلی بخش دیگری آگاه باشد. با قرارداد، هر بخش فقط باید بداند طرف مقابل چه وعده‌ای داده است.

## فرمت قرارداد

قرارداد یک فایل JSON است با نام `contract.json`. هسته این فایل را همزمان با مانیفست می‌خواند و اعتبارسنجی می‌کند.

## ساختار یک قرارداد

هر قرارداد از سه بخش تشکیل می‌شود:

**name** — شناسه یکتای این قرارداد در سیستم. هیچ دو قراردادی نمی‌توانند نام یکسان داشته باشند.

**version** — شماره نسخه قرارداد به فرمت MAJOR.MINOR. هر بار که قرارداد به شکلی تغییر کند که سازگاری را بشکند، نسخه باید افزایش یابد.

**methods** — فهرست متدهای عمومی که این بخش ارائه می‌دهد. هر متد باید نام، پارامترهای ورودی، و نوع خروجی مشخصی داشته باشد.

## اعتبارسنجی قرارداد

هسته اعتبارسنجی قرارداد را در دو مرحله انجام می‌دهد:

**مرحله اول — اعتبارسنجی ساختار** (قبل از بارگذاری بخش):
```text
آیا فیلدهای name، version، و methods وجود دارند؟
آیا نسخه فرمت درستی دارد؟ (MAJOR.MINOR)
آیا هر متد نام، پارامترها، و نوع خروجی دارد؟
```

**مرحله دوم — اعتبارسنجی پیاده‌سازی** (بعد از بارگذاری بخش):
```text
آیا همه متدهای اعلام‌شده پیاده‌سازی شده‌اند؟
آیا امضای هر متد با قرارداد مطابقت دارد؟
```

اگر اعتبارسنجی ساختار شکست بخورد، بخش بارگذاری نمی‌شود. اگر اعتبارسنجی پیاده‌سازی شکست بخورد، بخش از رجیستری حذف می‌شود و هسته خطای واضحی اعلام می‌کند.

## نمونه قرارداد افزونه

```json
{
  "name": "my-company.contract.storage",
  "version": "1.0",
  "methods": [
    {
      "name": "get",
      "params": [
        { "name": "key", "type": "string" }
      ],
      "returns": "any"
    },
    {
      "name": "set",
      "params": [
        { "name": "key", "type": "string" },
        { "name": "value", "type": "any" }
      ],
      "returns": "boolean"
    },
    {
      "name": "delete",
      "params": [
        { "name": "key", "type": "string" }
      ],
      "returns": "boolean"
    }
  ]
}
```

## نمونه قرارداد ماژول

```json
{
  "name": "my-company.contract.task",
  "version": "1.0",
  "methods": [
    {
      "name": "getAll",
      "params": [],
      "returns": "Task[]"
    },
    {
      "name": "getById",
      "params": [
        { "name": "id", "type": "string" }
      ],
      "returns": "Task"
    },
    {
      "name": "create",
      "params": [
        { "name": "data", "type": "TaskInput" }
      ],
      "returns": "Task"
    },
    {
      "name": "update",
      "params": [
        { "name": "id", "type": "string" },
        { "name": "data", "type": "TaskInput" }
      ],
      "returns": "Task"
    },
    {
      "name": "delete",
      "params": [
        { "name": "id", "type": "string" }
      ],
      "returns": "boolean"
    }
  ]
}
```

## قوانین قرارداد

### تعریف قرارداد

* هر ماژول، افزونه، پلتفرم، و رابط کاربری باید یک فایل `contract.json` داشته باشد
* قرارداد باید پیش از پیاده‌سازی نوشته شود
* قرارداد متعلق به بخشی است که آن را تعریف کرده، نه بخشی که از آن استفاده می‌کند
* هسته همیشه قرارداد و مانیفست را با هم می‌خواند؛ نبود هر کدام خطاست

### تغییر قرارداد

* تغییری که متد جدید اضافه کند بدون تغییر متدهای موجود، نسخه فرعی را افزایش می‌دهد
* تغییری که امضای متد موجود را بشکند، نسخه اصلی را افزایش می‌دهد
* هیچ بخشی نمی‌تواند قرارداد بخش دیگری را تغییر دهد

## آنچه قرارداد نیست

* قرارداد شامل جزئیات پیاده‌سازی نمی‌شود
* قرارداد توضیح نمی‌دهد چطور کاری انجام می‌شود، فقط چه کاری انجام می‌شود
* قرارداد شامل وابستگی‌ها، پیکربندی، یا رویدادها نمی‌شود؛ اینها در مانیفست تعریف می‌شوند
