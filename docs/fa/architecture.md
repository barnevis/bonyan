# معماری

## مقدمه

این معماری یک ساختار ماژولار و چندسکویی برای توسعه محصولات نرم‌افزاری مبتنی بر JavaScript، HTML و CSS است که بدون وابستگی به چارچوب‌های سنگین طراحی شده است.

هدف اصلی این معماری کاهش پیچیدگی، افزایش قابلیت نگهداری، بهبود استفاده مجدد از کد، و ساده‌سازی توسعه توسط انسان و مدل‌های هوش مصنوعی است.

این معماری تلاش می‌کند وابستگی به چارچوب‌ها، رفتارهای پنهان، و انتزاع‌های پیچیده را کاهش دهد و در عوض بر قراردادهای شفاف، مرزبندی مشخص، و ساختارهای قابل‌پیش‌بینی تکیه کند.

هر بخش از سیستم یک قرارداد دارد که مشخص می‌کند چه متدهایی ارائه می‌دهد و چه پیام‌هایی منتشر می‌کند. بخش‌ها یکدیگر را فقط از طریق این قراردادها می‌شناسند و هیچ بخشی از جزئیات داخلی بخش دیگری آگاه نیست.

## اهداف معماری

معماری باید:

* توسعه محصول را سریع‌تر کند
* استفاده مجدد از قابلیت‌ها را افزایش دهد
* وابستگی میان بخش‌های محصول را کاهش دهد
* توسعه توسط انسان و مدل‌های هوش مصنوعی را ساده‌تر و قابل‌اعتمادتر کند
* در بلندمدت بدون بازنویسی کامل تکامل پیدا کند

## ساختار کلی

این معماری پیام‌محور است. هیچ بخشی مستقیماً با بخش دیگری در ارتباط نیست. تنها راه ارتباط میان بخش‌ها از طریق مجراهایی است که توسط هسته مدیریت می‌شوند.

سامانه از شش بخش اصلی تشکیل شده است که هر کدام مسئولیت مشخصی دارند:

```text
                    هسته
                      |
             مجراها
      ┌─────────┬──────┴──────┬─────────┐
  ماژول‌ها  افزونه‌ها  رابط کاربری  پلتفرم
      └─────────┴─────────────┴─────────┘
                   قراردادها
```

* **هسته** چرخه حیات سیستم را مدیریت می‌کند، بخش‌ها را بارگذاری می‌کند، و دسترسی به مجراها را در اختیار آن‌ها قرار می‌دهد
* **مجراها** تنها راه ارتباط میان همه بخش‌ها هستند — هیچ استثنایی وجود ندارد
* **قراردادها** توافق رسمی میان بخش‌ها هستند و مشخص می‌کنند هر بخش چه متدهایی ارائه می‌دهد و چه پیام‌هایی منتشر می‌کند
* **ماژول‌ها** منطق و قابلیت‌های اصلی محصول را پیاده‌سازی می‌کنند
* **افزونه‌ها** زیرساخت‌های قابل استفاده مشترک را فراهم می‌کنند
* **رابط کاربری** داده را نمایش می‌دهد، تعاملات کاربر را دریافت می‌کند، و داده‌های وارد شده توسط کاربر را جمع‌آوری می‌کند
* **پلتفرم** اتصال به محیط اجرا را فراهم می‌کند

## لایه‌های معماری

### هسته

هسته بخش مرکزی معماری است که مسئول هماهنگی و مدیریت اجزای معماری است.

هسته منطق پروژه را نمی‌شناسد، وابسته به رابط کاربری نیست، و از جزئیات محیط اجرا آگاه نیست. هسته در انتقال داده میان بخش‌ها دخالتی ندارد.

وظایف هسته:

* مدیریت چرخه حیات سیستم
* بارگذاری و مدیریت چرخه حیات بخش‌ها
* اعتبارسنجی مانیفست و قرارداد
* رجیستری سرویس‌ها
* مدیریت مجراها
* مدیریت خطا
* نگهداری حالت مشترک
* مدیریت پیکربندی
* لاگ داخلی

### مجراها

مجراها تنها راه ارتباط میان بخش‌ها هستند. هیچ بخشی نمی‌تواند مستقیماً با بخش دیگری ارتباط برقرار کند — نه فراخوانی مستقیم، نه اشتراک‌گذاری حالت، نه وابستگی مستقیم. اگر بخشی می‌خواهد اطلاع‌رسانی کند، داده منتقل کند، یا درخواستی داشته باشد، باید این کار را با انتشار پیام در یک مجرا انجام دهد.

این قانون استثنا ندارد.

در سطح مجرا، معماری میان «رویداد»، «درخواست»، «فرمان» یا «پاسخ» تفاوت فنی قائل نمی‌شود. هر چیزی که از مجرا عبور می‌کند یک **پیام** است. معنای پیام از طریق `type` پیام و قرارداد بخش‌ها مشخص می‌شود.

#### انواع مجرا

```text
مجرای سیستم   ← پیام‌های عمومی که همه بخش‌های مجاز می‌توانند بشنوند
مجرای خصوصی  ← پیام‌های خصوصی هر بخش که فقط بخش‌های مجاز می‌توانند بشنوند
```

مجرای خصوصی فقط برای داده‌های حساس یا پیام‌های پرتکرار داخلی استفاده می‌شود که ربطی به بقیه بخش‌ها ندارند. دسترسی به مجرای خصوصی هر بخش در `bootstrap.json` تعریف می‌شود.

#### عملیات مجرا

```text
انتشار  ← یک بخش پیامی را در یک مجرا منتشر می‌کند
اشتراک ← یک بخش اعلام می‌کند که به نوع مشخصی از پیام اهمیت می‌دهد
لغو    ← یک بخش اعلام می‌کند که دیگر به آن نوع پیام گوش نمی‌دهد
```

ناشر نمی‌داند چه کسی پیام را دریافت می‌کند. دریافت‌کننده نمی‌داند چه کسی پیام را منتشر کرده. این جدایی آگاهانه است و بخش‌ها را از هم مستقل نگه می‌دارد.

#### ساختار پیام

هر پیام باید یک `type` داشته باشد. پیام می‌تواند `data` و `correlationId` هم داشته باشد. مجرا یا هسته هنگام انتشار، اطلاعات فنی پیام را کامل می‌کند.

ساختار پایه پیام:

```json
{
  "id": "msg-123",
  "type": "task:created",
  "source": "my-company.module.task",
  "data": {
    "id": "task-1"
  },
  "correlationId": "req-456",
  "timestamp": "2026-06-11T12:00:00.000Z"
}
```

فیلدها:

* `id` شناسه یکتای پیام است و توسط مجرا یا هسته تولید می‌شود
* `type` نوع پیام است و باید در قرارداد مربوط تعریف شده باشد
* `source` نام بخشی است که پیام را منتشر کرده و توسط مجرا یا هسته تعیین می‌شود
* `data` داده پیام است و اختیاری است
* `correlationId` برای تطبیق پیام‌های مرتبط استفاده می‌شود و اختیاری است
* `timestamp` زمان انتشار پیام است و توسط مجرا یا هسته تعیین می‌شود

قوانین داده در پیام:

* داده‌های کوچک می‌توانند مستقیماً در پیام منتقل شوند
* داده‌های حجیم باید در یک افزونه ذخیره‌سازی موقت نگه داشته شوند و فقط شناسه آن در پیام منتقل شود
* داده‌های حساس مثل توکن یا رمز عبور نباید در مجرای سیستم قرار بگیرند

```text
task:created       →  { data: { id: "123" } }
file:uploaded      →  { data: { id: "abc123", size: 10485760, mimeType: "image/png" } }
transactions:ready →  { data: { ref: "store-key-xyz" } }
core:ready         →  بدون data
```

#### تطبیق پیام‌های مرتبط

هر پیامی که پاسخ یا ادامه آن باید با پیام اصلی تطبیق داده شود می‌تواند `correlationId` داشته باشد. پیام‌های بعدی باید همان `correlationId` را برگردانند تا فرستنده بتواند آن‌ها را به هم مرتبط کند.

```text
report:get        → { correlationId: "req-123", data: { from: "2024-01-01", to: "2024-12-31" } }
report:get:result → { correlationId: "req-123", data: { result: {...} } }
report:get:failed → { correlationId: "req-123", data: { code: "REPORT_FAILED" } }
```

هر بخشی که پیامی منتشر می‌کند و انتظار پاسخ دارد، مسئول مدیریت حالت عدم پاسخ است. اگر در بازه زمانی مورد انتظار پاسخی نرسید، همان بخش باید خطا را مدیریت کند.

#### مسئولیت‌های حداقلی مجرا

مجرا در نسخه‌های اولیه معماری باید دست‌کم این مسئولیت‌ها را داشته باشد:

* انتشار پیام در مجرای مجاز
* اشتراک روی نوع مشخصی از پیام
* لغو اشتراک هنگام توقف بخش
* کنترل دسترسی بخش‌ها به مجراها
* تکمیل اطلاعات فنی پیام مانند `id`، `source`، و `timestamp`
* جلوگیری از اثر گذاشتن خطای یک شنونده بر شنونده‌های دیگر
* ثبت تلاش برای دسترسی غیرمجاز به مجرا

### افزونه‌ها

افزونه‌ها زیرساخت‌ها و قابلیت‌های مشترک میان پروژه‌ها را فراهم می‌کنند. افزونه‌ها پیش از ماژول‌ها راه‌اندازی می‌شوند تا زیرساخت مورد نیاز ماژول‌ها از ابتدا در دسترس باشد.

افزونه‌ها مسئول فراهم‌کردن زیرساخت هستند، نه پیاده‌سازی منطق محصول.

نمونه‌ها:

* ذخیره‌سازی
* احراز هویت
* مسیریابی
* ارتباط با هوش مصنوعی
* تحلیل داده

### ماژول‌ها

ماژول‌ها ویژگی‌ها و منطق اصلی محصول را پیاده‌سازی می‌کنند. هر ماژول نماینده یک قابلیت کسب‌وکار یا ویژگی مستقل است.

نمونه‌ها:

* مدیریت وظایف
* گفت‌وگو
* پروژه‌ها
* اعلان‌ها

ماژول‌ها می‌توانند از افزونه‌ها استفاده کنند، اما نباید مستقیماً به یکدیگر وابسته باشند. هر ماژول فقط از طریق مجراها با بقیه سیستم در ارتباط است.

### رابط کاربری

لایه رابط کاربری مسئول نمایش داده، دریافت تعاملات کاربر، و جمع‌آوری داده‌های وارد شده توسط کاربر است.

رابط کاربری نباید شامل منطق اصلی باشد. منطق کسب‌وکار باید خارج از رابط کاربری قرار گیرد تا قابلیت آزمایش، استفاده مجدد، و اجرا در محیط‌های مختلف داشته باشد.

تمام ارتباطات رابط کاربری از طریق مجراها انجام می‌شود:

```text
ماژول پیام منتشر می‌کند  →  رابط کاربری مطلع می‌شود  →  داده را از طریق یک مجرا می‌خواهد  →  نمایش به‌روز می‌شود
```

### پلتفرم

لایه پلتفرم مسئول اتصال محصول به محیط اجرا است و امکان اجرای محصول در محیط‌های مختلف را فراهم می‌کند.

نمونه‌ها:

* مرورگر
* دسکتاپ
* موبایل

تمام دسترسی‌ها به قابلیت‌های محیط اجرا باید از طریق قرارداد پلتفرم انجام شود. هیچ بخشی نباید مستقیماً با محیط اجرا ارتباط برقرار کند.

## اصول بنیادین

### ۱. سادگی بر پیچیدگی مقدم است

سادگی یک انتخاب تزئینی نیست، بلکه یک اصل معماری است. هر بخش از محصول باید قابل‌درک، رفتار قابل‌پیش‌بینی، و بدون نیاز به دانش پنهان قابل توسعه باشد.

پیچیدگی تنها زمانی پذیرفته می‌شود که مزیت واقعی و بلندمدت ایجاد کند.

### ۲. منطق کسب‌وکار باید مستقل باشد

منطق محصول نباید به رابط کاربری، نوع پایگاه داده، محیط اجرا یا سرویس‌های خارجی وابسته باشد و باید بتواند بدون تغییر در محیط‌های مختلف اجرا شود.

### ۳. همه‌چیز باید از طریق قراردادها ارتباط برقرار کند

اجزای محصول نباید مستقیماً به پیاده‌سازی یکدیگر وابسته باشند. ارتباط میان بخش‌ها باید تنها از طریق قراردادهای مشخص و پایدار انجام شود.

قراردادها پایه اصلی توسعه‌پذیری، قابلیت تعویض، آزمون‌پذیری، و توسعه توسط هوش مصنوعی هستند.

### ۴. هسته باید کوچک و ناآگاه باقی بماند

هسته نباید منطق پروژه، رابط کاربری، یا جزئیات محیط اجرا را بشناسد. هرچه هسته کوچک‌تر و عمومی‌تر باشد، محصول پایدارتر و قابل‌استفاده‌تر خواهد بود.

### ۵. همه قابلیت‌ها باید قابل‌تعویض باشند

هیچ بخش از محصول نباید به یک پیاده‌سازی خاص وابسته شود. سیستم ذخیره‌سازی، احراز هویت، موتور رابط کاربری، یا روش ارتباط با API باید بدون تغییر در منطق اصلی قابل جایگزینی باشند.

### ۶. ماژول‌ها باید مستقل و ایزوله باشند

هر ماژول باید مسئولیت مشخصی داشته باشد، مرز واضحی تعریف کند، و بدون وابستگی مستقیم به سایر ماژول‌ها توسعه یابد.

### ۷. رابط کاربری فقط لایه نمایش است

وظیفه رابط کاربری نمایش داده، دریافت تعامل کاربر، جمع‌آوری داده‌های وارد شده توسط کاربر، و ارسال پیام‌هاست. تمام تصمیم‌ها و قوانین اصلی باید خارج از رابط کاربری قرار بگیرند.

### ۸. معماری باید برای انسان و هوش مصنوعی قابل‌فهم باشد

این معماری برای مدل‌های هوش مصنوعی نیز باید قابل‌تحلیل و قابل‌توسعه باشد. به همین دلیل محصول باید ساختارهای واضح، فایل‌های کوچک، وابستگی محدود، رفتارهای شفاف، و قراردادهای مشخص داشته باشد.

### ۹. ایزوله‌سازی خطا

خرابی یا نقص در یک ماژول یا افزونه نباید باعث از کار افتادن کل سیستم شود. خطاها باید در همان مرزی که رخ می‌دهند مهار شوند و سیستم باید بتواند به کار خود، حتی با کاهش قابلیت، ادامه دهد.

اگر یک بخش در طول اجرا دچار خطای بحرانی شود و از رجیستری حذف شود، بخش‌های وابسته به کار ادامه می‌دهند. این بخش‌ها باید به پیام `core:part-failed` گوش بدهند و قابلیت‌های وابسته به بخش از دست رفته را غیرفعال کنند.

### ۱۰. مالکیت صریح داده‌ها

هر قطعه از داده در سیستم باید یک مالک مشخص و واحد داشته باشد. هیچ بخشی نباید داده بخش دیگر را به صورت مستقیم تغییر دهد. درخواست برای تغییر داده باید از طریق قراردادها و پیام‌ها انجام شود.

### ۱۱. انتقال داده باید از طریق مجراها انجام شود

داده میان بخش‌ها فقط از طریق مجراها منتقل می‌شود. هیچ بخشی نمی‌تواند داده بخش دیگری را مستقیماً بخواند یا تغییر دهد. اگر داده کوچک بود می‌تواند مستقیماً در پیام منتقل شود. اگر داده حجیم بود باید در افزونه ذخیره‌سازی موقت نگه داشته شود و فقط شناسه آن در پیام منتقل شود. تشخیص حجیم بودن داده بر عهده توسعه‌دهنده است؛ اصل کلی اینست که اگر انتقال داده از طریق پیام باعث مشکل عملکردی شود، باید به روش ذخیره‌سازی و انتقال شناسه تغییر کند.

## قوانین ارتباط میان بخش‌ها

* هسته نباید به افزونه‌ها، ماژول‌ها، یا پروژه خاص وابسته باشد
* همه بخش‌ها فقط از طریق مجراها با یکدیگر ارتباط برقرار می‌کنند
* بخش‌ها یکدیگر را فقط از طریق قراردادها می‌شناسند
* هیچ بخشی نباید مستقیماً به پیاده‌سازی داخلی بخش دیگر وابسته باشد

## هدف نهایی

هدف این معماری ساخت محصولی است که توسعه آن سریع، نگهداری آن ساده، وابستگی کمی داشته باشد، در محیط‌های مختلف اجرا شود، و بتواند در بلندمدت بدون بازنویسی کامل تکامل پیدا کند.

این معماری تلاش می‌کند تعادلی میان سادگی، انعطاف‌پذیری، توسعه‌پذیری و پایداری بلندمدت ایجاد کند.

## اصول تست‌پذیری

تست‌پذیری در این معماری یک اصل است، نه یک انتخاب. معماری باید به گونه‌ای طراحی شود که هر بخش بتواند مستقل از بقیه سیستم تست شود.

### اصول پایه

* هر بخش باید بدون نیاز به هسته قابل تست باشد
* هر بخش باید بتوان وابستگی‌هایش را با نمونه‌های جعلی جایگزین کرد
* قراردادها پایه تست‌پذیری هستند؛ اگر قرارداد مشخص باشد، تست نوشتن ساده است
* منطق داخلی هر بخش باید مستقل از محیط اجرا قابل تست باشد

### تست قرارداد

هر پیاده‌سازی باید در برابر قرارداد خود تست شود. این یعنی:

* همه متدهای اعلام‌شده در قرارداد باید تست داشته باشند
* تست‌ها باید رفتار اعلام‌شده را، نه جزئیات پیاده‌سازی را، بررسی کنند
* اگر قرارداد تغییر کند، تست‌ها باید به‌روز شوند

## وابستگی به کتابخانه‌های خارجی

وابستگی به کتابخانه‌های خارجی باید کنترل‌شده و محدود باشد. هر کتابخانه خارجی یک وابستگی است که نگهداری، به‌روزرسانی، و جایگزینی آن را پیچیده‌تر می‌کند.

قوانین وابستگی برای هر بخش:

**هسته** — هیچ کتابخانه خارجی نباید داشته باشد. هسته باید کاملاً مستقل باشد تا در هر محیطی بدون وابستگی اضافه اجرا شود.

**افزونه‌ها** — می‌توانند کتابخانه خارجی داشته باشند چون زیرساخت هستند و مسئول پیاده‌سازی قابلیت‌های تخصصی‌اند. وابستگی‌های خارجی افزونه باید در مانیفست اعلام شوند.

**ماژول‌ها** — نباید مستقیماً به کتابخانه خارجی وابسته باشند. اگر ماژولی به قابلیتی از یک کتابخانه خارجی نیاز دارد، باید آن قابلیت از طریق یک افزونه فراهم شود.

**پلتفرم** — می‌تواند کتابخانه خارجی داشته باشد چون به قابلیت‌های محیط اجرا وابسته است.

**رابط کاربری** — می‌تواند کتابخانه‌های نمایشی داشته باشد، اما نباید از کتابخانه‌هایی استفاده کند که منطق کسب‌وکار دارند.

این قوانین از سه اصل معماری پیروی می‌کنند: استقلال بخش‌ها، قابلیت تعویض، و کوچک نگه داشتن هسته.
