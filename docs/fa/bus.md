# گذرگاه رویداد

## مقدمه

گذرگاه رویداد تنها مسیر ارتباطی میان بخش‌های معماری است. هیچ ماژول، افزونه، پلتفرم یا رابط کاربری نباید بخش دیگری را مستقیم فراخوانی کند یا به حالت داخلی آن دسترسی داشته باشد.

هدف گذرگاه، کاهش وابستگی مستقیم، ساده‌کردن جایگزینی بخش‌ها، و قابل‌پیش‌بینی کردن جریان ارتباطات برای انسان و مدل‌های هوش مصنوعی است.

## اصل اصلی

در سطح گذرگاه، معماری میان رویداد، درخواست، فرمان یا پاسخ تفاوت فنی قائل نمی‌شود.

```text
هر چیزی که از گذرگاه عبور می‌کند، پیام است.
```

معنای پیام از طریق دو چیز مشخص می‌شود:

* `type` پیام
* قراردادی که پیام را تعریف کرده است

## انواع گذرگاه

```text
system.bus      ← مجرای عمومی سیستم
part-name.bus   ← مجرای خصوصی یک بخش
```

گذرگاه سیستم برای پیام‌های عمومی استفاده می‌شود. گذرگاه خصوصی برای داده‌های حساس یا پیام‌های پرتکراری استفاده می‌شود که به همه سیستم مربوط نیستند.

## عملیات گذرگاه

گذرگاه فقط سه عملیات پایه دارد:

```text
publish     ← انتشار پیام
subscribe   ← گوش دادن به یک نوع پیام
unsubscribe ← لغو گوش دادن
```

هر ابزار کمکی دیگر، مانند request/response helper، باید در نهایت با همین عملیات پایه کار کند.

## ساختار پیام

هر پیام باید `type` داشته باشد. بقیه فیلدها بر اساس نیاز پیام اضافه می‌شوند.

```json
{
  "id": "msg-123",
  "type": "storage:get",
  "source": "my-company.module.task",
  "data": {
    "store": "tasks",
    "key": "task-1"
  },
  "correlationId": "req-456",
  "timestamp": "2026-06-11T12:00:00.000Z"
}
```

## فیلدهای پیام

| فیلد | اجباری | توضیح |
| --- | --- | --- |
| `id` | بله | شناسه یکتای پیام، ساخته‌شده توسط گذرگاه یا هسته |
| `type` | بله | نوع پیام، تعریف‌شده در قرارداد |
| `source` | بله | نام بخش منتشرکننده، تعیین‌شده توسط گذرگاه یا هسته |
| `data` | خیر | داده پیام |
| `correlationId` | خیر | شناسه مشترک برای مرتبط کردن چند پیام |
| `timestamp` | بله | زمان انتشار پیام، تعیین‌شده توسط گذرگاه یا هسته |

بخش منتشرکننده فقط لازم است `type` و در صورت نیاز `data` و `correlationId` را ارائه کند. فیلدهای `id`، `source` و `timestamp` توسط گذرگاه یا هسته تکمیل می‌شوند.

## تطبیق پیام‌های مرتبط

اگر یک بخش پیامی منتشر کند و انتظار پاسخ یا ادامه جریان داشته باشد، باید از `correlationId` استفاده کند.

```text
storage:get        → correlationId: req-1
storage:get:result → correlationId: req-1
storage:get:failed → correlationId: req-1
```

گذرگاه مسئول تصمیم‌گیری درباره timeout، retry، یا نبود پاسخ نیست. این مسئولیت با بخشی است که پیام اولیه را منتشر کرده است. گذرگاه می‌تواند ابزار کمکی فراهم کند، اما آن ابزار هم باید با پیام‌های عادی کار کند.

## قواعد انتشار

* هر پیام باید روی مجرایی منتشر شود که بخش منتشرکننده به آن دسترسی دارد
* پیام بدون `type` معتبر نیست
* داده حساس نباید روی `system.bus` منتشر شود
* داده حجیم باید در افزونه ذخیره‌سازی موقت قرار گیرد و فقط شناسه آن در پیام منتقل شود
* ناشر نباید بداند چه بخش‌هایی پیام را دریافت می‌کنند

## قواعد اشتراک

* هر بخش فقط می‌تواند روی مجراهایی اشتراک بگیرد که در `bootstrap.json` برای آن مجاز شده است
* هر اشتراک باید روی یک `type` مشخص انجام شود
* هر اشتراک باید در زمان توقف بخش لغو شود
* دریافت‌کننده نباید به منبع پیام اعتماد کامل داشته باشد و باید داده ورودی را اعتبارسنجی کند

## ایزوله‌سازی خطا

خطای یک شنونده نباید باعث شکست تحویل پیام به شنونده‌های دیگر شود.

اگر یک handler هنگام پردازش پیام خطا بدهد:

```text
۱. خطا در همان handler مهار می‌شود
۲. خطا ثبت می‌شود
۳. تحویل پیام به سایر handlerها ادامه پیدا می‌کند
۴. در صورت نیاز، پیام خطای استاندارد منتشر می‌شود
```

## کنترل دسترسی

هسته مسئول ساخت مجراها و تعیین دسترسی بخش‌ها به آن‌هاست. اگر بخشی تلاش کند به مجرایی دسترسی پیدا کند که برای آن مجاز نیست:

```text
۱. درخواست رد می‌شود
۲. تلاش غیرمجاز ثبت می‌شود
۳. سیستم به کار خود ادامه می‌دهد
```

## نام‌گذاری پیام‌ها

قالب کلی پیام‌ها:

```text
نام:فعل
نام:فعل:result
نام:فعل:failed
```

نمونه:

```text
storage:get
storage:get:result
storage:get:failed

task:create
task:create:result
task:create:failed
task:created
```

## آنچه گذرگاه نیست

* گذرگاه محل نگهداری منطق کسب‌وکار نیست
* گذرگاه محل نگهداری داده محصول نیست
* گذرگاه نباید درباره معنای تجاری پیام‌ها تصمیم بگیرد
* گذرگاه جایگزین اعتبارسنجی ورودی در بخش‌ها نیست
* گذرگاه نباید باعث شود بخش‌ها مسئولیت مدیریت timeout و خطاهای خود را فراموش کنند
