# رابط کاربری (UI Layer)

## مقدمه

لایه رابط کاربری مسئول نمایش داده به کاربر و دریافت تعامل از اوست. رابط کاربری هیچ دانشی از منطق محصول، قوانین کسب‌وکار، یا زیرساخت ندارد. این جداسازی تضمین می‌کند که تغییر در ظاهر یا فناوری رابط کاربری به منطق محصول آسیب نمی‌زند و رابط کاربری می‌تواند بدون تغییر در افزونه‌ها جایگزین شود.

رابط کاربری افزونه نیست اما مانند افزونه‌ها یک مانیفست دارد که هسته آن را در زمان راه‌اندازی می‌خواند.

## مانیفست رابط کاربری

مانیفست رابط کاربری بخشی از معماری بنیان است چون مرز دسترسی رابط کاربری را مشخص می‌کند. هسته فقط به سرویس‌هایی که در مانیفست UI اعلام شده‌اند دسترسی می‌دهد و چیز دیگری در اختیار رابط کاربری قرار نمی‌دهد.

مانیفست رابط کاربری مشخص می‌کند UI به کدام سرویس‌ها نیاز دارد و چه رویدادهایی منتشر می‌کند. ساختار آن مشابه مانیفست افزونه است اما سبک‌تر:

```json
{
  "name": "myapp.ui",
  "version": "1.0.0",
  "dependencies": {
    "required": [
      "myapp.transactions.service",
      "myapp.accounts.service"
    ],
    "optional": [
      "myapp.reports.service"
    ]
  },
  "events": [
    {
      "name": "ui:save-transaction-requested",
      "description": "کاربر درخواست ذخیره یک تراکنش داده",
      "data": [
        { "name": "requestId", "type": "string" }
      ]
    },
    {
      "name": "ui:filter-changed",
      "description": "کاربر فیلتر نمایش را تغییر داده",
      "data": [
        { "name": "filterId", "type": "string" }
      ]
    }
  ]
}
```

رابط کاربری بخش `provides` ندارد چون سرویس ارائه نمی‌دهد. بخش `type` هم ندارد چون افزونه نیست.

## نقطه ورود

هسته در زمان راه‌اندازی به نقطه ورود رابط کاربری دسترسی پیدا می‌کند. نقطه ورود همان سه چیزی را دریافت می‌کند که افزونه‌ها دریافت می‌کنند: سرویس‌های اعلام‌شده در `dependencies` مانیفست UI، امکان انتشار و گوش‌دادن به رویداد از طریق گذرگاه رویداد، و پیکربندی نهایی رابط کاربری.

شکل دقیق این تبادل، شامل نام فایل‌ها و امضای توابع، جزئیات پیاده‌سازی است و در سند بهترین شیوه‌ها آمده است، نه در این سند.

## مسیر اصلی عملیات کاربر

این مسیر مشخص می‌کند وقتی کاربر عملی انجام می‌دهد، داده و کنترل دقیقاً چطور بین رابط کاربری، سرویس، و افزونه جریان پیدا می‌کند.

**برای خواندن داده:** رابط کاربری مستقیماً سرویس اعلام‌شده در مانیفستش را فراخوانی می‌کند و داده را برای نمایش می‌گیرد.

**برای ارسال درخواست کاربر:** رابط کاربری داده‌ای که کاربر وارد کرده را مستقیماً به متد مربوطه در سرویس اعلام‌شده می‌فرستد. منطق اصلی همیشه داخل سرویس افزونه اجرا می‌شود، حتی وقتی رابط کاربری مستقیماً آن سرویس را صدا می‌زند. رابط کاربری هیچ منطقی را خودش اجرا نمی‌کند.

**بعد از انجام عملیات:** افزونه‌ای که سرویس را ارائه داده، پس از انجام موفق عملیات، یک رویداد دامنه منتشر می‌کند تا رابط کاربری یا بخش‌های دیگر از تغییر باخبر شوند. انتشار رویداد دامنه مسئولیت افزونه است، نه رابط کاربری.

**رویداد عملیات کاربر چه زمانی استفاده می‌شود:** رویداد عملیات کاربر فقط برای اعلام تغییرات یا درخواست‌های سبک رابط کاربری است، مثلاً تغییر فیلتر نمایش یا اعلام اینکه کاربر فرمی را باز کرده. این رویداد جایگزین فراخوانی مستقیم سرویس نیست و برای انتقال داده کامل فرم یا اجرای منطق اصلی استفاده نمی‌شود.

## مسئولیت‌های رابط کاربری

رابط کاربری چهار مسئولیت دارد:

**نمایش داده:** برای نمایش داده، مستقیماً سرویس مربوطه را فراخوانی می‌کند. رابط کاربری می‌تواند داده را برای نمایش قالب‌بندی کند اما داده اصلی را تغییر نمی‌دهد.

**دریافت تعامل کاربر:** کلیک، اسکرول، انتخاب، و سایر تعاملات کاربر را دریافت می‌کند.

**دریافت داده از کاربر:** داده‌ای که کاربر وارد می‌کند را دریافت می‌کند و مستقیماً به سرویس مربوطه می‌فرستد.

**انتشار رویداد عملیات کاربر:** برای اعلام تغییرات یا درخواست‌های سبک رابط کاربری، رویداد مناسب را با پیشوند `ui:` منتشر می‌کند. مثل همه رویدادها در بنیان، این رویداد فقط داده شناسایی‌کننده حمل می‌کند، نه payload کامل و نه داده حساس. اگر رویداد یک شناسه حمل می‌کند، داده کامل مرتبط با آن شناسه باید از قبل در یک سرویس اعلام‌شده ثبت شده باشد تا گیرنده بتواند آن را بازیابی کند.

## آنچه رابط کاربری نیست

**رابط کاربری مسئول منطق محصول نیست.** قوانین کسب‌وکار، محاسبات، و اعتبارسنجی‌های مربوط به محصول در افزونه‌ها هستند، حتی اگر رابط کاربری مستقیماً سرویس افزونه را صدا بزند.

**رابط کاربری مسئول نگهداری داده محصول نیست.** داده متعلق به افزونه‌های محصول است.

**رابط کاربری درباره قواعد محصول و محتوای معتبر داده تصمیم نمی‌گیرد.** اینکه چه داده‌ای معتبر است، چه قانونی روی آن حاکم است، و چه چیزی باید نمایش داده شود، تصمیمی است که افزونه محصول می‌گیرد.

**رابط کاربری مسئول زیرساخت نیست.** ذخیره‌سازی، مسیریابی، و سایر سرویس‌های پایه به رابط کاربری تعلق ندارند.

**رابط کاربری قوانین محصول را نمی‌داند.** این قوانین در افزونه محصول هستند.

## مرز نمایش و منطق

**مجاز است:** قالب‌بندی نمایشی مانند تبدیل تاریخ به فارسی، نمایش عدد با جداکننده هزار، یا تغییر رنگ بر اساس مقدار. همچنین مرتب‌کردن یا فیلتر کردن نمایشی که فقط نحوه نمایش را تغییر می‌دهد، نه داده اصلی را.

**ممنوع است:** فیلتر کردن واقعی داده، محاسبه، اعتبارسنجی قوانین محصول، یا هر تصمیمی که به منطق کسب‌وکار مربوط می‌شود.

## پیشنهاد پیاده‌سازی

بنیان هیچ فناوری خاصی را برای رابط کاربری تحمیل نمی‌کند. انتخاب فناوری به توسعه‌دهنده است.

با این حال، استفاده از Web Component پیشنهاد می‌شود. Web Component یک استاندارد مرورگر است، به هیچ فریم‌ورکی وابسته نیست، و بدون تغییر در Capacitor و Tauri هم کار می‌کند.
