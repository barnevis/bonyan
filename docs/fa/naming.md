# نام‌گذاری (Naming)

## مقدمه

قراردادهای نام‌گذاری در بنیان ساده و کم هستند. هدف این است که هر کسی که کد را می‌خواند بلافاصله بفهمد با چه چیزی طرف است و از کجا آمده.

## نام‌گذاری افزونه

همه افزونه‌ها از یک فرمت دو بخشی پیروی می‌کنند:

```
vendor.name
```

**vendor** نام شخص، تیم، یا سازمانی است که افزونه را نوشته. این بخش از تداخل نام بین افزونه‌های مختلف جلوگیری می‌کند.

**name** نام کوتاه و توصیفی افزونه است. باید بیانگر کار افزونه باشد.

نمونه‌ها:
- `myfinance.transactions`
- `myfinance.accounts`
- `myfinance.storage`
- `acme.reports`

هیچ دو افزونه‌ای در یک سیستم نمی‌توانند نام یکسان داشته باشند. هسته در زمان راه‌اندازی این یکتایی را بررسی می‌کند.

## نام‌گذاری سرویس

سرویس‌ها همان نام افزونه‌ای را دارند که آنها را فراهم می‌کند به علاوه پسوند `.service`:

```
vendor.name.service
```

نمونه‌ها:
- `myfinance.transactions.service`
- `myfinance.storage.service`
- `acme.reports.service`

## نام‌گذاری رویداد

در بنیان دو نوع رویداد وجود دارد که باید از هم تفکیک شوند.

### رویداد دامنه (Domain Event)

رویدادی که افزونه منتشر می‌کند تا اعلام کند چیزی در سیستم اتفاق افتاده است. این رویدادها در زمان گذشته نوشته می‌شوند و از فرمت زیر پیروی می‌کنند:

```
pluginname:eventname
```

**pluginname** نام بخش دوم نام افزونه است، بدون vendor.

**eventname** باید بیانگر چیزی باشد که اتفاق افتاده، نه چیزی که باید انجام شود.

نمونه‌ها:
- `transactions:saved`
- `transactions:deleted`
- `accounts:updated`
- `storage:ready`
- `core:plugin-failed`
- `core:plugin-crashed`

### رویداد عملیات کاربر (UI Action Event)

رویدادی که رابط کاربری منتشر می‌کند تا اعلام کند کاربر تغییری سبک در رابط کاربری داده است، مثل تغییر فیلتر یا باز شدن یک پنل. این رویداد جایگزین فراخوانی مستقیم سرویس نیست و هرگز برای اجرای منطق اصلی محصول مثل ذخیره یا حذف استفاده نمی‌شود؛ آن عملیات همیشه از طریق فراخوانی مستقیم سرویس انجام می‌شوند. این رویدادها با پیشوند `ui:` شروع می‌شوند و نام عمل را به صورت اسم می‌آورند:

```
ui:action-requested
```

نمونه‌ها:
- `ui:filter-changed`
- `ui:panel-opened`
- `ui:view-mode-changed`

تفاوت مهم این است که رویداد دامنه می‌گوید «این اتفاق افتاد» و رویداد عملیات کاربر می‌گوید «یک درخواست سبک یا تغییر حالت رابط کاربری رخ داده». رویداد عملیات کاربر هرگز جایگزین فراخوانی مستقیم سرویس برای عملیات اصلی محصول نمی‌شود.

مثل رویدادهای دامنه، رویدادهای عملیات کاربر هم فقط داده شناسایی حمل می‌کنند، نه داده کامل، و هرگز داده حساس مانند رمز عبور یا توکن همراه ندارند.

## قوانین کلی نام‌گذاری

**حروف کوچک و خط تیره.** همه نام‌ها با حروف کوچک نوشته می‌شوند. اگر نام چند کلمه دارد از خط تیره استفاده می‌شود. مثال: `myfinance.budget-reports`.

**نام‌های توصیفی.** نام باید بگوید افزونه یا رویداد چیست، نه چطور کار می‌کند.

**اختصار ممنوع است.** `transactions` درست است، `txn` اشتباه است. خوانایی مهم‌تر از کوتاهی است.

**رویدادهای دامنه در زمان گذشته.** رویداد دامنه چیزی است که اتفاق افتاده. `saved` درست است، `save` اشتباه است. استثنا رویدادهای هسته هستند که با `core:` شروع می‌شوند.

**رویدادهای عملیات کاربر با پیشوند `ui:`.** همه رویدادهایی که رابط کاربری منتشر می‌کند با `ui:` شروع می‌شوند و با `-changed`، `-opened`، یا الگوهای مشابه ختم می‌شوند که تغییر حالت رابط کاربری را نشان می‌دهند، نه اجرای عملیات اصلی محصول.
