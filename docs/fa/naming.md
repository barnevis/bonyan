# نام‌گذاری

## مقدمه

نام‌گذاری یکپارچه و قراردادی یکی از مهم‌ترین عوامل خوانایی، نگهداری، و توسعه‌پذیری این معماری است. وقتی نام‌گذاری‌ها پیش‌بینی‌پذیر باشند، هم انسان و هم مدل‌های هوش مصنوعی بدون نیاز به توضیح اضافه می‌توانند نوع، مالک، و نقش هر بخش را بفهمند.

## اصول کلی

* همه نام‌ها به انگلیسی نوشته می‌شوند
* برای نام‌های چند بخشی از `kebab-case` استفاده می‌شود
* نقطه جداکننده بخش‌های اصلی نام است
* دونقطه جداکننده نام پیام از فعل و وضعیت آن است
* هیچ دو بخشی در یک سیستم نمی‌توانند نام یکسان داشته باشند

## قالب نام‌گذاری بخش‌ها

همه بخش‌های اصلی معماری از یک قالب سه‌بخشی پیروی می‌کنند:

```text
سازنده.نوع.نام
```

* **سازنده** — نام شخص، تیم، یا سازمانی که این بخش را نوشته
* **نوع** — نوع بخش در معماری
* **نام** — نام مشخص این بخش

### هسته

```text
my-company.core.main
```

هسته معمولاً یکتاست، اما اگر چند تیم هسته‌های متفاوتی داشته باشند، نام سازنده آن‌ها را از هم تفکیک می‌کند.

### ماژول

```text
my-company.module.task
my-company.module.chat
my-company.module.notification
```

### افزونه

```text
my-company.plugin.indexed-db
my-company.plugin.auth
my-company.plugin.router
```

### پلتفرم

```text
my-company.platform.browser
my-company.platform.mobile
my-company.platform.desktop
```

### رابط کاربری

```text
my-company.ui.web
my-company.ui.mobile
my-company.ui.desktop
```

### قرارداد

```text
my-company.contract.task
my-company.contract.storage
my-company.contract.auth
```

## نام‌گذاری عناصر داخلی

عناصر داخلی چون در محدوده یک بخش مشخص هستند، نیازی به نام سازنده ندارند.

### مجرا

مجراها از قالب `نام.channel` پیروی می‌کنند:

```text
system.channel       ← مجرای سیستم، مشترک میان همه بخش‌ها
task.channel         ← مجرای خصوصی ماژول task
indexed-db.channel   ← مجرای خصوصی افزونه indexed-db
```

### پیام‌ها

در سطح مجرا، همه چیز پیام است. معماری میان رویداد، درخواست، فرمان یا پاسخ تفاوت فنی قائل نمی‌شود. معنای پیام از طریق نام پیام و قرارداد مشخص می‌شود.

پیام‌ها از قالب `نام:فعل` پیروی می‌کنند:

```text
نام:فعل
```

برای پیام‌هایی که نتیجه یا شکست یک عملیات را اعلام می‌کنند، از پسوندهای ساده استفاده می‌شود:

```text
نام:فعل:result  ← نتیجه موفق یک عملیات
نام:فعل:failed  ← شکست یک عملیات
```

نمونه‌ها:

```text
task:create
task:create:result
task:create:failed
task:created
task:updated
task:deleted
storage:get
storage:get:result
storage:get:failed
storage:synced
auth:login
auth:login:result
auth:login:failed
auth:logged-in
```

قانون پیشنهادی برای خوانایی:

```text
پیام‌هایی که یک کار را آغاز می‌کنند معمولاً فعل حال دارند: task:create، storage:get
پیام‌هایی که وقوع یک اتفاق را اعلام می‌کنند معمولاً فعل گذشته دارند: task:created، auth:logged-in
پیام‌های نتیجه و شکست از :result و :failed استفاده می‌کنند
```

### متدها (عملیات‌ها)

متدها از قالب `نام.فعل` یا `نام.فعلمفعول` پیروی می‌کنند:

```text
task.getAll
task.getById
task.create
task.update
task.delete
```

### پیام‌های هسته

پیام‌هایی که هسته در مجرای سیستم منتشر می‌کند از قالب `core:فعل` پیروی می‌کنند:

```text
core:ready          ← سیستم آماده است
core:part-failed    ← یک بخش با خطای بحرانی مواجه شد
core:part-loaded    ← یک بخش با موفقیت بارگذاری شد
core:part-unloaded  ← یک بخش از رجیستری حذف شد
```

## قوانین نام‌گذاری

* نام سازنده باید مشخص و یکتا باشد تا تعارض نام میان بخش‌های مختلف پیش نیاید
* نوع بخش باید دقیقاً یکی از مقادیر `core`، `module`، `plugin`، `platform`، یا `ui` باشد
* نام بخش باید کوتاه، مشخص، و توصیفی باشد
* از اختصارات ناشناخته پرهیز شود
* نام پیام باید به اندازه کافی مشخص باشد که بدون توضیح اضافه قابل فهم باشد
