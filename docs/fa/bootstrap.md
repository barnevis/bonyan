# پیکربندی راه‌اندازی

## مقدمه

پیکربندی راه‌اندازی تنها منبع معتبری است که هسته از آن می‌فهمد یک پروژه از چه بخش‌هایی تشکیل شده و دسترسی بخش‌ها به مجراها چگونه تنظیم می‌شود. هسته بدون این فایل نمی‌داند چه افزونه‌ها، ماژول‌ها، پلتفرم‌ها، و رابط‌های کاربری باید بارگذاری شوند.

هر پروژه یک فایل پیکربندی راه‌اندازی دارد. این فایل اولین چیزی است که هسته پس از اجرا می‌خواند.

## فرمت پیکربندی راه‌اندازی

پیکربندی راه‌اندازی یک فایل JSON است با نام `bootstrap.json`.

## ساختار پیکربندی راه‌اندازی

پیکربندی راه‌اندازی از هشت بخش تشکیل می‌شود:

**architecture** — نسخه معماری‌ای که این پروژه بر اساس آن ساخته شده.

**core** — نام و نسخه هسته و افزونه‌هایی که هسته مستقیماً به آن‌ها نیاز دارد.

**platform** — پلتفرمی که این پروژه در آن اجرا می‌شود، همراه با مسیر پوشه آن تا هسته بتواند فایل‌های `manifest.json` و `contract.json` آن را بخواند.

**plugins** — فهرست افزونه‌هایی که باید بارگذاری شوند، به ترتیب اولویت بارگذاری. هر افزونه علاوه بر نام و نسخه، باید مسیر پوشه خود را نیز مشخص کند.

**modules** — فهرست ماژول‌هایی که باید بارگذاری شوند. هر ماژول علاوه بر نام و نسخه، باید مسیر پوشه خود را نیز مشخص کند.

**ui** — رابط کاربری این پروژه، همراه با مسیر پوشه آن.

**channels** — قوانین دسترسی به مجراها. توسعه‌دهنده بر اساس نیاز و امنیت پروژه تعیین می‌کند چه بخش‌هایی می‌توانند به هر مجرا دسترسی داشته باشند.

هر مجرا می‌تواند با یکی از دو سیاست تعریف شود:

* **allow** — فقط بخش‌های نام‌برده‌شده می‌توانند به این مجرا دسترسی داشته باشند. بقیه بخش‌ها دسترسی ندارند.
* **deny** — همه بخش‌ها می‌توانند به این مجرا دسترسی داشته باشند به جز بخش‌های نام‌برده‌شده. اگر `deny` خالی باشد، همه بخش‌ها دسترسی دارند.

**سیاست پیش‌فرض:** اگر یک مجرا در بخش `channels` تعریف نشده باشد، دسترسی به آن **بسته** است — هیچ بخش خارجی نمی‌تواند به آن دسترسی داشته باشد. توسعه‌دهنده باید هر مجرایی که نیاز به دسترسی خارجی دارد را صریحاً در `channels` تعریف کند.

**config** — مقادیر پیکربندی برای همه بخش‌ها که بر مقادیر پیش‌فرض مانیفست اولویت دارند.

## نمونه پیکربندی راه‌اندازی

```json
{
  "architecture": "1.1",

  "core": {
    "name": "my-company.core.main",
    "version": "1.2.0",
    "plugins": [
      {
        "name": "my-company.plugin.logger",
        "version": "1.0.0"
      }
    ]
  },

  "platform": {
    "name": "my-company.platform.browser",
    "version": "1.0.0",
    "path": "./platform"
  },

  "plugins": [
    {
      "name": "my-company.plugin.logger",
      "version": "1.0.0",
      "path": "./plugins/logger"
    },
    {
      "name": "my-company.plugin.indexed-db",
      "version": "2.1.0",
      "path": "./plugins/indexed-db"
    },
    {
      "name": "my-company.plugin.auth",
      "version": "1.0.0",
      "path": "./plugins/auth"
    },
    {
      "name": "my-company.plugin.router",
      "version": "1.3.0",
      "path": "./plugins/router"
    }
  ],

  "modules": [
    {
      "name": "my-company.module.task",
      "version": "1.0.0",
      "path": "./modules/task"
    },
    {
      "name": "my-company.module.chat",
      "version": "1.2.0",
      "path": "./modules/chat"
    },
    {
      "name": "my-company.module.notification",
      "version": "1.0.0",
      "path": "./modules/notification"
    }
  ],

  "ui": {
    "name": "my-company.ui.web",
    "version": "1.0.0",
    "path": "./ui/web"
  },

  "config": {
    "my-company.plugin.indexed-db": {
      "db-name": "my-app-db"
    },
    "my-company.plugin.logger": {
      "level": "info"
    },
    "my-company.ui.web": {
      "theme": "dark",
      "language": "fa"
    }
  },

  "channels": {
    "system.channel": {
      "deny": []
    },
    "my-company.ui.web.channel": {
      "allow": [
        "my-company.module.task",
        "my-company.module.notification"
      ]
    },
    "my-company.module.task.channel": {
      "deny": [
        "my-company.ui.mobile"
      ]
    }
  }
}
```

## فرآیند راه‌اندازی

هسته پس از خواندن فایل پیکربندی، مراحل زیر را به ترتیب انجام می‌دهد:

۱.  بررسی نسخه معماری
۲.  بارگذاری افزونه‌های مورد نیاز هسته
۳.  در صورت وجود افزونه لاگ، انتقال لاگ از لاگ داخلی به آن
۴.  بارگذاری پلتفرم تعریف‌شده در bootstrap.json
۵.  اعتبارسنجی مانیفست و قرارداد پلتفرم
۶.  بارگذاری پلتفرم
۷.  بارگذاری افزونه‌ها به ترتیب فهرست
۸.  اعتبارسنجی مانیفست و قرارداد هر افزونه
۹.  بررسی وابستگی‌های هر افزونه (نبود وابستگی اختیاری خطا نیست)
۱۰. بارگذاری ماژول‌ها
۱۱. اعتبارسنجی مانیفست و قرارداد هر ماژول
۱۲. بررسی وابستگی‌های هر ماژول (نبود وابستگی اختیاری خطا نیست)
۱۳. بارگذاری رابط کاربری تعریف‌شده در bootstrap.json
۱۴. اعتبارسنجی مانیفست و قرارداد رابط کاربری
۱۵. ساخت مجراهای تعریف‌شده در بخش channels؛ اگر بخشی در مانیفست مجرای خصوصی خواسته اما در channels تعریف نشده باشد، هسته آن را با دسترسی پیش‌فرض می‌سازد
۱۶. تنظیم دسترسی مجراها بر اساس بخش channels
۱۷. راه‌اندازی رابط کاربری
۱۸. سیستم آماده است

رفتار هسته در برابر خطاهای راه‌اندازی به نوع بخش بستگی دارد:

* اگر هسته، پلتفرم اصلی، یا رابط کاربری اصلی قابل راه‌اندازی نباشد، راه‌اندازی سیستم متوقف می‌شود
* اگر یک افزونه یا ماژول به وابستگی ضروری دسترسی نداشته باشد، همان بخش بارگذاری نمی‌شود
* اگر نبود یک بخش باعث شود بخش‌های وابسته نتوانند کار کنند، آن بخش‌های وابسته نیز بارگذاری نمی‌شوند
* نبود وابستگی اختیاری باعث توقف راه‌اندازی نمی‌شود
* همه خطاهای راه‌اندازی باید با پیام واضح در لاگ ثبت شوند

هدف این رفتار آن است که سیستم تا حد ممکن با قابلیت کاهش‌یافته اجرا شود، اما اگر بخش‌های پایه‌ای سیستم در دسترس نباشند، راه‌اندازی به شکل شفاف متوقف شود.

## هسته و مانیفست

هسته برخلاف سایر بخش‌ها فایل manifest.json ندارد. اطلاعات هسته مستقیماً از بخش `core` در bootstrap.json خوانده می‌شود. این استثنا به این دلیل است که هسته پیش از هر مکانیزم بارگذاری‌ای اجرا می‌شود و نمی‌تواند منتظر خواندن مانیفست خودش بماند.

## متغیرهای محیطی

مقادیر حساس مثل کلید API، رمز پایگاه داده، و توکن‌های سرویس نباید در `bootstrap.json` به صورت متن ساده ذخیره شوند. این مقادیر باید از طریق متغیرهای محیطی تأمین شوند.

برای استفاده از متغیر محیطی در `bootstrap.json`، از قالب `${ENV_VAR_NAME}` استفاده می‌شود:

```json
{
  "config": {
    "my-company.plugin.auth": {
      "api-key": "${AUTH_API_KEY}",
      "secret": "${AUTH_SECRET}"
    }
  }
}
```

هسته در زمان راه‌اندازی این مقادیر را از محیط اجرا می‌خواند و جایگزین می‌کند. همه مقادیر جایگزین‌شده به صورت رشته هستند — هسته نوع داده را حفظ نمی‌کند. هر بخش مسئول تبدیل مقادیر پیکربندی به نوع مورد نیاز خود است. اگر یک متغیر محیطی که `required: true` دارد در محیط اجرا تعریف نشده باشد، هسته راه‌اندازی را متوقف می‌کند و خطای واضحی اعلام می‌کند.

## قوانین پیکربندی راه‌اندازی

* هر پروژه فقط یک فایل `bootstrap.json` دارد
* این فایل فقط توسط هسته خوانده می‌شود
* هیچ ماژول یا افزونه‌ای نمی‌تواند مستقیماً به این فایل دسترسی داشته باشد
* ترتیب افزونه‌ها در فهرست، ترتیب بارگذاری آن‌هاست
* افزونه‌هایی که هسته به آن‌ها نیاز دارد باید در فهرست plugins هم وجود داشته باشند
* اگر یک بخش در فهرست نباشد، هسته آن را بارگذاری نمی‌کند
* هر پروژه فقط یک پلتفرم و یک رابط کاربری دارد
* مقادیر config در این فایل بر مقادیر پیش‌فرض مانیفست اولویت دارند
* پیکربندی پس از راه‌اندازی سیستم تغییر نمی‌کند
* وابستگی‌های اختیاری در زمان راه‌اندازی بررسی می‌شوند اما نبودشان خطا نیست
* بخش channels فقط قوانین دسترسی بخش‌ها به مجراها را مشخص می‌کند؛ مسیریابی یا تفسیر پیام‌ها مسئولیت هسته نیست
