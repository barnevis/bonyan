# Channel

## Introduction

The Channel is the only communication path between architectural parts. No module, plugin, platform, or user interface should directly call another part or access its internal state.

The goal of the channel is to reduce direct dependencies, simplify replacing parts, and make communication flows predictable for both humans and AI models.

## Core Principle

At the channel level, the architecture does not technically distinguish between an event, request, command, or response.

```text
Everything that passes through the channel is a message.
```

The meaning of a message is defined by two things:

* The message `type`
* The contract that defines the message

## Channel Types

```text
system.channel      ← The general system channel
part-name.channel   ← The private channel of a part
```

The system channel is used for general messages. A private channel is used for sensitive data or high-frequency messages that are not relevant to the whole system.

## Channel Operations

The channel has only three basic operations:

```text
publish     ← Publish a message
subscribe   ← Listen to a message type
unsubscribe ← Stop listening
```

Any helper utility, such as a request/response helper, must ultimately work through these same basic operations.

## Message Structure

Every message must have a `type`. Other fields are added based on the needs of the message.

```json
{
  "id": "msg-123",
  "type": "storage:get",
  "source": "my-company.module.task",
  "data": {
    "store": "tasks",
    "key": "task-1"
  },
  "correlationId": "req-456",
  "timestamp": "2026-06-11T12:00:00.000Z"
}
```

## Message Fields

| Field | Required | Description |
| --- | --- | --- |
| `id` | Yes | Unique message identifier, generated by the channel or the Core |
| `type` | Yes | Message type, defined in the contract |
| `source` | Yes | Name of the publishing part, set by the channel or the Core |
| `data` | No | Message payload |
| `correlationId` | No | Shared identifier for relating multiple messages |
| `timestamp` | Yes | Message publish time, set by the channel or the Core |

The publishing part only needs to provide `type` and, if needed, `data` and `correlationId`. The `id`, `source`, and `timestamp` fields are completed by the channel or the Core.

## Matching Related Messages

If a part publishes a message and expects a response or continuation, it must use `correlationId`.

```text
storage:get        → correlationId: req-1
storage:get:result → correlationId: req-1
storage:get:failed → correlationId: req-1
```

The channel is not responsible for deciding timeouts, retries, or missing responses. That responsibility belongs to the part that published the original message. The channel may provide helper utilities, but those helpers must still work through normal messages.

## Publishing Rules

* Every message must be published on a channel the publishing part is allowed to access.
* A message without `type` is invalid.
* Sensitive data must not be published on `system.channel`.
* Large data must be stored in a temporary storage plugin, and only its identifier should be transferred in the message.
* The publisher must not know which parts receive the message.

## Subscription Rules

* Each part can only subscribe to channels that are allowed for it in `bootstrap.json`.
* Each subscription must target a specific `type`.
* Every subscription must be removed when the part stops.
* The receiver must not fully trust the message source and must validate incoming data.

## Error Isolation

An error in one listener must not cause message delivery to fail for other listeners.

If a handler throws an error while processing a message:

```text
1. The error is contained within that handler
2. The error is logged
3. Message delivery continues for other handlers
4. If needed, a standard error message is published
```

## Access Control

The Core is responsible for creating channels and defining part access to them. If a part attempts to access a channel it is not allowed to access:

```text
1. The request is rejected
2. The unauthorized attempt is logged
3. The system continues operating
```

## Message Naming

General message format:

```text
name:verb
name:verb:result
name:verb:failed
```

Examples:

```text
storage:get
storage:get:result
storage:get:failed

task:create
task:create:result
task:create:failed
task:created
```

## What the Channel is Not

* The channel is not a place for business logic.
* The channel is not a place for product data.
* The channel should not decide the business meaning of messages.
* The channel is not a replacement for input validation inside parts.
* The channel should not make parts forget their own responsibility for timeouts and errors.
