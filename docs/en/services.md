# Services

## Introduction

A service is a capability that one plugin provides and other plugins can use. Services are the only formal mechanism for request-response interaction between plugins.

All services are provided by plugins. The Core provides no services. The Core only holds the addresses of services in the Registry.

## Service Types

Services are categorized by the type of plugin that provides them:

**Infrastructure services** are provided by infrastructure plugins and are available before all other plugins. Other plugins can rely on these services being present. Examples: StorageService, RouterService, NotificationService, StateService.

**Product services** are provided by product plugins and expose the application's core business logic to others. Example: TransactionService registered by the transactions plugin and used by the reports plugin.

**Feature services** are provided by feature plugins and expose optional capabilities to others. Example: SearchService registered by the search plugin.

**Platform services** are provided by platform plugins and enable compatibility with the runtime environment. Example: FileSystemService implemented by the appropriate plugin depending on the platform.

**Transport services** are provided by transport plugins and manage message transfer with the outside world. Examples: HttpService, WebSocketService.

## Registry

The Registry is the index of addresses for all registered services. When a plugin registers a service, the Registry stores its address. When another plugin needs that service, the Registry returns the address.

The Registry is a discovery point, not a communication intermediary. The call happens directly between the requesting plugin and the service. The Core plays no role in the call itself.

## Service Registration Cycle

### Registration

A plugin registers its services in the Registry at startup. To register a service, the plugin must have declared it in the `provides` section of its manifest. A service not declared in the manifest cannot be registered.

### Access

A plugin that needs a service must declare it in the `dependencies` section of its manifest. The Core performs this check once at startup and gives the plugin the addresses of its declared services. From that point on, calls are direct and the Core is not involved. Any attempt to access an undeclared service is rejected and logged.

### Deregistration

When a plugin is deactivated, its services are removed from the Registry. Plugins that had a required dependency on those services are notified via the Event Bus.

## Service versus Event Bus

These two communication mechanisms must not be confused. The boundary between them is precise:

**Service** is used when a plugin needs something and expects a response. Example: the reports plugin asks TransactionService for the total transactions of this month.

**Event Bus** is used when a plugin announces that something happened and does not wait for a response. Example: the transactions plugin announces that a new transaction was saved.

If a response is needed, use a service. If it is just a notification, use the Event Bus.

## Event Bus

### Event Structure

Every event has a defined structure. The plugin sends only two fields and the Core adds the rest:

**The plugin sends:**

```json
{
  "type": "transactions:saved",
  "data": { "transactionId": "tx-456" }
}
```

**The Core completes and publishes:**

```json
{
  "id": "evt-123",
  "type": "transactions:saved",
  "source": "myfinance.transactions",
  "data": { "transactionId": "tx-456" },
  "timestamp": "2026-06-28T10:30:00Z"
}
```

**id:** A unique identifier for the event. Generated by the Core.

**type:** The event name in `pluginname:eventname` format. Defined by the plugin.

**source:** The name of the publishing plugin. Taken by the Core from the plugin's manifest. The plugin cannot forge this value.

**data:** Minimum identifying information. Defined by the plugin. Rules are in the next section.

**timestamp:** The time the event was published. Added by the Core.

### Rules for the data field

The `data` field is for identifying information only, not for transferring complete data. The listener uses this information to call the relevant service if it needs the full data.

Allowed: identifiers the listener needs to call the relevant service.

Not allowed: the complete content of an entity, lists of entities, sensitive data such as passwords and tokens, personal user information.

### Publishing an event

A plugin publishes an event. Events a plugin publishes must be declared in its manifest. Event names follow the `pluginname:eventname` format. Naming details are in `naming.md`.

### Listening to an event

Any plugin may listen to any event. Listening to an event requires no special access and does not need to be declared in the manifest.

## Service Security

The Core checks once at startup that a plugin has declared the services it needs in its manifest. Any runtime attempt to access an undeclared service is rejected and recorded in the Core's internal log.

Full details are in `security.md`.
